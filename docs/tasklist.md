## Итерации

### 1. Подготовка локального запуска
Цель: обеспечить предсказуемый запуск приложения на локальной машине, чтобы любая следующая итерация начиналась с "зелёного старта".
Чек-лист:
- [x] `make worker` запускает процесс и корректно завершает при корректной конфигурации
- [x] Ошибки конфигурации понятны и не выглядят как "краш без объяснений"
- [x] В корне есть `.env.example` с обязательными env (включая `DB_*`) для корректного старта `make worker`

### 2. База и хранение данных
Цель: подготовить хранение, чтобы приложение могло сохранять и читать данные для синхронизации.
Чек-лист:
- [x] В MySQL доступны нужные структуры данных для синхронизации (таблицы созданы заранее вручную)
- [x] Воркeр успешно соединяется с БД и делает базовое чтение `avito_accounts` без ошибок
- [x] `make worker` завершает работу с кодом `0` при успешном чтении `avito_accounts`

### 3. Поток "аккаунт -> последний отчет"
Цель: добавить первый рабочий сценарий получения данных от Avito и запись "последнего отчёта".
Чек-лист:
- [x] Для одного аккаунта фиксируется факт получения последнего отчёта в БД
- [x] Повторный запуск этой итерации не ломает процесс (даже если данных уже нет новых)

### 4. Идемпотентность повторных запусков
Цель: сделать повторы безопасными: повторный запуск не плодит дубликаты и не повторяет то, что уже сделано.
Чек-лист:
- [x] 2 последовательных запуска дают предсказуемый результат без дублей
- [x] Ошибки от повторных запусков не мешают обработке следующего аккаунта

### 5. Поток "отчет -> ошибочные объявления"
Цель: расширить синхронизацию до уровня "ошибочные объявления", связанных с конкретным отчётом.
Чек-лист:
- [x] Для нового отчёта сохраняется набор ошибочных объявлений
- [x] Повторы на "старом" отчёте не создают дубликаты error-ads

### 6. OAuth: получение и сохранение токена
Цель: реализовать `OAuthClient`, который получает `access_token` через `client_credentials` и сохраняет его в `avito_accounts`.
Чек-лист:
- [x] Класс `OAuthClient` делает `POST OAUTH_TOKEN_URL` с `grant_type=client_credentials`, `client_id`, `client_secret`
- [x] Из ответа берётся `access_token` и `expires_in`; вычисляется `token_expires_at = NOW() + expires_in - 60s`
- [x] `access_token` и `token_expires_at` сохраняются в `avito_accounts` (upsert по `id`)
- [x] `make worker`: после запуска в `avito_accounts` видны заполненные `access_token` и `token_expires_at`

### 7. OAuth: управление жизненным циклом токена
Цель: не тратить запросы на обновление валидного токена и корректно обрабатывать протухший токен при HTTP-ошибках.
Чек-лист:
- [x] Перед запросом к Avito API проверяется: `access_token` непустой и `token_expires_at > NOW()` — если да, токен не перезапрашивается
- [x] Если `access_token` пустой или истёк — вызывается `OAuthClient` для получения нового токена
- [x] 2 запуска подряд с валидным токеном: в логах нет обращений к `OAUTH_TOKEN_URL`

### 8. Реальный запрос последнего отчёта
Цель: заменить `AVITO_FAKE_LAST_REPORT_ID` реальным `report_id`, получаемым от Avito API.
Чек-лист:
- [x] Если Avito API вернул 403 с `{"result": {"status": false, "message": "invalid access token"}}` — токен обновляется однократно и запрос повторяется; повторный 403 — ошибка аккаунта
- [x] Класс `AvitoApiClient` выполняет `GET AVITO_BASE_URL + AVITO_LAST_REPORT_PATH` с заголовком `Authorization: Bearer <access_token>`
- [x] Класс `ReportParser` (или метод) маппит JSON-ответ: поле `report_id` → `report_external_id` в DTO
- [x] `Worker` использует реальный `report_external_id` вместо значения из `AVITO_FAKE_LAST_REPORT_ID`
- [x] Переменная `AVITO_FAKE_LAST_REPORT_ID` удалена из кода и `.env.example`
- [x] `make worker`: в `avito_reports` появляется запись с реальным `report_id` от Avito

### 9. Реальный запрос ошибочных объявлений с пагинацией
Цель: заменить `AVITO_FAKE_ERROR_AD_IDS` реальными данными из Avito API; реализовать обход всех страниц пагинации.
Чек-лист:
- [x] `AvitoApiClient` запрашивает `GET AVITO_BASE_URL + AVITO_ERROR_ADS_PATH` с `{report_id}` подставленным в путь
- [x] `ErrorAdsParser` обходит все страницы (`page` от `0` до `meta.pages - 1`) в цикле
- [x] Из каждого элемента `items[]` берётся `ad_id` → `ad_external_id`; первый `messages[].title` с `type=error` → `error_type` (null если таких нет)
- [x] `Worker` использует реальные данные вместо `AVITO_FAKE_ERROR_AD_IDS`
- [x] Переменная `AVITO_FAKE_ERROR_AD_IDS` удалена из кода и `.env.example`
- [x] `make worker`: в `avito_error_ads` реальные `ad_id` и `error_type` из Avito

### 10. Фильтрация объявлений по коду ошибки
Цель: сохранять в `avito_error_ads` только те объявления, у которых хотя бы одно `messages[].code` входит в список `AVITO_ERROR_CODES`.
Чек-лист:
- [x] Новая env-переменная `AVITO_ERROR_CODES` (коды через запятую, например `1073,1504`; необязательная)
- [x] Если `AVITO_ERROR_CODES` не задан или пустой — ни одно объявление не сохраняется
- [x] Объявление сохраняется если хотя бы одно `messages[].code` входит в список
- [x] `error_type` — `messages[].title` первого сообщения с кодом из списка
- [x] `make worker`: в `avito_error_ads` только объявления с заданными кодами

### 11. Надежность обработки ошибок
Цель: гарантировать устойчивость процесса: ошибка в одном аккаунте не останавливает весь воркер.
Чек-лист:
- [x] Если обработка одного аккаунта падает, остальные аккаунты продолжают обрабатываться
- [x] Завершение процесса отражает итог: видно, был ли частичный фейл

### 12. Атомарная запись отчёта и объявлений
Цель: отчёт не должен оставаться в БД без объявлений — запись отчёта и объявлений выполняется в одной транзакции.
Чек-лист:
- [x] HTTP-запрос за `error_ads` выполняется ДО начала транзакции (транзакция не держит соединение во время HTTP)
- [x] `saveReport` и `saveMany` обёрнуты в `beginTransaction` / `commit`
- [x] При любом исключении внутри транзакции — `rollBack`; отчёт не сохраняется
- [x] Повторный запуск после неудачи обрабатывает отчёт заново (т.к. он не был закоммичен)

### 13. Обновление схемы БД
Цель: привести схему в соответствие с актуальными требованиями.
Чек-лист:
- [x] `refresh_token` удалён из `avito_accounts` (DROP COLUMN)
- [x] `avito_error_ads` получает `status ENUM('NEW','COMPLETED') NOT NULL DEFAULT 'NEW'`
- [x] `avito_error_ads` получает `rx_good_items INT NOT NULL` — число до первого `-` в `ad_external_id`
- [x] `sql/01_create_tables.sql` содержит финальную схему (без `refresh_token`, с новыми полями, уникальный ключ по `ad_external_id`)
- [x] `AvitoErrorAdsRepository::saveMany` заполняет `status='NEW'` и `rx_good_items`
- [x] `make worker`: новые поля видны в `avito_error_ads`

### 14. Логирование и эксплуатация
Цель: сделать систему пригодной к ежедневному запуску: удобная проверка запуска, понятные логи, отсутствие секретов в выводе.
Чек-лист:
- [x] Логи содержат полезный контекст (без секретов) и их удобно читать
- [x] Успешный запуск в Docker через `make worker` без отличий по смыслу от ожидаемого

### 15. CI/CD: публикация образа в GHCR
Цель: при каждом коммите в main автоматически собирать Docker-образ и публиковать его в GitHub Container Registry.
Чек-лист:
